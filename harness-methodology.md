# Harness 方法论：AI Agent 工程协作的自反馈治理框架

---

## 一句话定义

**Harness = 文档约束 + 自反馈闭环**

前者负责不跑偏，后者负责不盲写。

---

## 要解决的问题

AI Agent 参与编码时，最常见的三种失效模式：

1. **跑偏** — Agent 不了解项目约束，写出技术栈错误、命名不统一、架构违规的代码
2. **盲写** — Agent 只能确认"代码看起来对"，但无法验证"功能在真实环境里是否生效"
3. **自欺** — Agent 自己出题自己验，覆盖不到真实业务的边界数据和异常场景

Harness 框架通过三个 Part 分别解决这三个问题。

## 体系全景

```

│                                                                        │

│  │  执行计划 — 多文件变更有序可追溯                                  │  │
│  └──────────────────────────────────────────────────────────────────┘  │
```

## Part 1：文档约束

### 核心思想

Agent 不是人，不能靠"入职培训"和"口头提醒"。所有约束必须以 Agent 可解析的 Markdown 文档存在，并且通过自动注入机制确保不遗漏。

### 1.1 导引入口（AGENTS.md）

项目根目录放一份 `AGENTS.md`，作为 Agent 进入项目的**唯一入口**。职责：

- 一句话定位项目（做什么、什么技术栈）
- 知识文档索引表（什么时候读哪份文档）
- 执行计划目录结构
- 核心约束速览（5~10 条最关键的红线）

设计原则：Agent 只读这一份文件就能知道接下来该看什么，不需要遍历整个仓库。

### 1.2 知识文档体系

建议按以下四份文档组织（可以更多，但不建议更少）：

| 文档 | 内容 | When to read |
|------|------|-------------|
| 项目元信息 | 技术栈、目录结构、构建运行方式 | 初次进入项目 |
| 核心架构 | 分层架构、数据模型、API 规范、状态机 | 需要理解系统设计 |
| 工程约束 | 版本锁定、编码禁忌、命名规范、业务硬规则 | 写代码之前必读 |
| 自反馈工作流 | 执行计划机制、Linter 使用、反馈闭环流程 | 动手改代码时参照 |

### 1.3 自动注入规则

利用 Agent 平台的规则注入能力（如 CatPaw 的 `.catpaw/rules/`、Cursor 的 `.cursorrules`），将关键约束设为每次对话自动生效。推荐至少三份规则：

| 规则 | 职责 |
|------|------|
| 技术约束规则 | 版本锁定、编码禁忌、命名规范 |
| 文档维护规则 | 改代码必须同步更新对应文档 |
| 执行计划规则 | 多文件变更必须先建计划 |

关键：设置 `alwaysApply: true`（或对应平台的等价机制），确保 Agent 无需记住"去看规则"。

### 1.4 静态 Linter

写一个项目专用的 lint 脚本，编码约束中的每一条"禁止"都应该有对应的检查项。设计要求：

- **可 Agent 执行** — 一行命令即可运行，输出人和 Agent 都能解析
- **区分 ERROR / WARN** — ERROR 必须修复（exit 非0），WARN 建议关注
- **秒级反馈** — 基于 grep / 正则 / AST 静态分析，不需要启动服务
- **闭环强制** — 代码改完必须跑 linter，零 ERROR 才算完成

检查项举例（按你的项目替换具体内容）：

| 类型 | 示例 |
|------|------|
| 依赖版本锁定 | 检查 pom.xml / package.json 中核心依赖版本是否符合约束 |
| 禁止导入 | 扫描 import 语句，禁止出现废弃包 |
| 只读表保护 | 检查审计表是否有 UPDATE/DELETE 操作 |
| 分层违规 | 检查 Controller 层是否包含事务注解或复杂逻辑 |
| 硬编码检查 | 检查是否有魔法数字代替枚举常量 |
| SQL 规范 | 检查字符集、引擎、索引命名 |

### 1.5 执行计划机制

当变更涉及多个文件（建议阈值 ≥3 个文件）时，必须先创建执行计划再动手：

```markdown
# 执行计划：<标题>

## 目标
一句话描述。

## 验收标准
- [ ] 标准1
- [ ] 标准2
- [ ] linter 零违规
- [ ] 编译/构建通过

## 步骤
1. ...
2. ...

## 执行日志
- [时间] 做了什么 / 遇到什么问题 / 如何解决
```

生命周期：`创建 (active/) → 执行中 → 完成 (completed/) + 追加 activeLog`

## Part 2：自反馈工作流

### 核心思想

编译通过 ≠ 功能生效。工程瓶颈很快会从"写代码"变成"验证结果"。Agent 必须能走到"环境里真的跑过，并且有日志证据"，才算完成。

### 2.1 完整验证链路

```
代码 → commit/push → 构建 → 测试环境部署 → 发真实请求 → 查日志/指标 → 判断结果
                                                                         ↓
                                                          不通过 → 修代码 → 重新构建部署
```

这条链路必须被写成文档（`自反馈工作流.md`），而不是只存在于聊天记录或人脑中。文档中要明确：

| 环节 | 需要钉死的信息 |
|------|--------------|
| 构建 | 构建命令、CI/CD 入口、构建产物位置 |
| 部署 | 目标环境、集群、部署命令或平台入口 |
| 测试 | API 测试工具/平台、测试账号、基础请求模板 |
| 日志 | 日志平台入口、关键搜索条件、如何定位到目标 pod/实例 |

### 2.2 执行元信息（唯一事实源）

每一轮需求执行，需要一份 `当前活跃需求执行说明.md`，钉死运行时元信息：

```markdown
# 当前活跃需求执行说明

## 状态
active

## 适用范围
本轮执行的需求名称 / feature 分支

## 运行时元信息
- 代码分支：feature/xxx
- 构建入口：[CI 链接 / 构建命令]
- 部署环境：[env / 集群 / namespace]
- 测试入口：[API 测试平台 / curl 模板]
- 日志入口：[日志平台 + 搜索条件]
- 接口契约版本：[IDL/OpenAPI spec 版本]

## 优先级规则
当其他文档与本文档冲突时，以本文档为准。

## 阻塞处理
- 构建失败 → 检查依赖 / 检查分支是否最新
- 部署失败 → 检查配额 / 检查镜像
- 测试失败 → 先核对请求参数与接口契约是否一致
```

这份文档的意义：把"运行时元信息"从聊天里的临时说明升级成**能统一读取的事实源**。Agent 不需要回翻聊天记录去找"上次你说部署到哪个环境来着"。

### 2.3 执行日志

要求每一步关键动作都记录到执行日志。收益：

- Agent 长时间运行不会丢失上下文
- 人不在时 Agent 可继续工作
- 人回来后扫一眼日志就知道当前进展

日志内容建议包含：时间、操作、结果（成功/失败）、关键输出/错误信息、下一步计划。

### 2.4 运行态纠偏的价值

自反馈闭环真正的价值不是帮你"更快确认自己是对的"，而是帮你更快发现自己到底错在哪。典型场景：

- **接口契约漂移** — 代码本身没错，但测试入口使用的 schema/契约版本与服务实际运行版本不一致，导致误判为"代码有 bug"
- **DDL 不对齐** — 代码中的字段名和生产环境实际表结构不一致，必须部署到真实环境 + 发真实请求才能暴露
- **配置缺失** — 功能开关 / 环境变量 / 中间件配置在目标环境未生效

这类问题都不可能靠"看代码感觉"发现，必须走完完整链路。

### 2.5 验收纪律

**不拿到运行态证据，就不允许把执行计划移到 `completed/`。**

"运行态证据"的最低标准：

- 在目标环境发过真实请求
- 有日志/响应体证明功能确实生效
- 证据被记录在执行日志中（可回溯）

## Part 3：对抗测试

### 核心思想

需求接口能跑通，不代表业务真的稳。线上实际业务数据还没打过来，单 case 通过不代表全场景通过。所以需要**大量覆盖测试**，而且测试集不应该由写代码的 Agent 自己出。

### 3.1 双 Agent 模式

```
┌─────────────────────────────────────────────┐
│  Agent A（对抗测试 Agent）                   │
│  角色：攻击者 / 出题者                       │
│  输入：接口文档 + 业务规则 + 数据模型        │
│  输出：对抗测试数据集 + 测试计划             │
└──────────────────────┬──────────────────────┘
                       ↓
┌─────────────────────────────────────────────┐
│  Agent B（需求 Agent）                       │
│  角色：开发者 / 解题者                       │
│  输入：测试计划 + 测试数据集                 │
│  动作：跑自反馈链路，遇 bug 修复重跑         │
│  验收：所有数据集通过                        │
└─────────────────────────────────────────────┘
```

关键设计：**出题者和解题者分离**，避免"自己验自己"的盲区。

### 3.2 对抗 Agent 的输出要求

对抗 Agent 应该生成：

- **测试数据集** — 覆盖正常路径、边界值、异常输入、并发场景、权限越界
- **测试计划** — 明确执行顺序、前置条件、期望结果、判定标准
- **覆盖度说明** — 标注每条 case 覆盖了哪条业务规则

### 3.3 对抗测试执行计划模板

```markdown
# 对抗测试执行计划：<接口/功能名称>

## 状态
待执行 / 执行中 / 完成

## 执行记录要求
- 阶段级进展 → activeLog.md
- case 级测试日志、请求响应、bug 记录 → activeTestLog.md

## 目的
验证 <功能> 在大量边界数据和异常场景下的稳定性。

## 优先级规则
当本文档与其他测试说明冲突时，以本文档为准。

## 数据集
| Case ID | 场景描述 | 输入 | 期望结果 | 实际结果 | 状态 |
|---------|---------|------|---------|---------|------|
| TC-001  | ...     | ...  | ...     |         | 待测 |
| TC-002  | ...     | ...  | ...     |         | 待测 |
```

### 3.4 四级验证层次

```
Level 1: 静态 Linter
  → 编码合规性：命名、禁止导入、分层规范
         ↓
Level 2: 编译/构建
  → 语法正确性、依赖完整性
         ↓
Level 3: 自反馈工作流
  → 单接口在真实环境跑通 + 日志证据
         ↓
Level 4: 对抗测试
  → 大量边界数据覆盖 + 多接口联动 + 异常场景
```

每一层解决不同的问题，跳过任何一层都会留下验证盲区。

## 快速上手清单

如果你要在自己的项目中落地 Harness 框架，按以下顺序执行：

### Step 1：建立文档骨架

```
your-project/
├── AGENTS.md                           ← 导引入口
├── docs/
│   ├── agents/
│   │   ├── 01-project-meta.md          ← 技术栈、目录、构建方式
│   │   ├── 02-architecture.md          ← 架构、数据模型、状态机
│   │   ├── 03-constraints.md           ← 工程约束（你的红线）
│   │   └── 04-workflow.md              ← 自反馈工作流定义
│   └── exec-plans/
│       ├── active/                     ← 当前执行计划
│       ├── completed/                  ← 归档
│       └── activeLog                   ← 时间线
├── .catpaw/rules/  (或 .cursorrules 等)
│   ├── project-constraints.md          ← 自动注入：技术约束
│   ├── doc-maintenance.md              ← 自动注入：文档维护
│   └── exec-plan-discipline.md         ← 自动注入：执行计划
└── scripts/
    └── lint-project.sh                 ← 项目专用 Linter
```

### Step 2：写约束文档

1. 列出你项目的版本硬约束（语言版本、框架版本、核心依赖）
2. 列出编码禁忌（禁止出现的 pattern）
3. 列出命名规范（数据库、代码、API 各一套）
4. 列出业务硬规则（数量限制、权限规则、数据完整性）
5. 列出状态流转规则（如果有状态机）

### Step 3：实现 Linter 脚本

把 Step 2 中每一条"禁止"转化为可执行的检查：

```bash
#!/usr/bin/env bash
# 项目 Harness Linter
set -euo pipefail
ERRORS=0

# 示例检查项：
# 1. 扫描禁止导入
# 2. 检查版本锁定
# 3. 检查只读表保护
# 4. 检查分层规范
# ...

if [ $ERRORS -gt 0 ]; then exit 1; fi
```

### Step 4：配置自动注入规则

将核心约束写成自动注入规则文件，确保 `alwaysApply: true`。

### Step 5：建立自反馈工作流

写一份 `自反馈工作流.md`，明确：

- 构建入口和命令
- 部署入口和目标环境
- 测试入口（API 测试平台 / curl / 测试框架）
- 日志入口（日志平台 / 本地日志路径）
- 验收标准：不拿到运行态证据不允许标记完成

### Step 6：首次实战验证

选一个中等规模的需求（涉及 3~5 个文件），走完整闭环：

1. 创建执行计划
2. 编码
3. 跑 Linter → 零 ERROR
4. 构建 → 部署 → 发请求 → 查日志
5. 记录执行日志
6. 归档执行计划

### Step 7：引入对抗测试（需求稳定后）

1. 开一个独立 Agent 会话，给它接口文档和业务规则
2. 让它生成对抗测试数据集
3. 回到需求 Agent，拿数据集跑自反馈
4. 遇 bug 修复，重跑，直到全部通过

## 设计原则

| 编号 | 原则 | 一句话解释 |
|------|------|-----------|
| P1 | 文档驱动 | 所有规则以 Markdown 存在，Agent 可直接解析 |
| P2 | 分层递进 | 从入口到细节，Agent 按需深入，不信息过载 |
| P3 | 自动注入 | 关键规则不依赖 Agent 记忆，系统强制注入 |
| P4 | 即时反馈 | Linter 提供秒级合规校验 |
| P5 | 强制纪律 | 多文件变更有序可追溯 |
| P6 | 文档同步 | 代码和文档在同一次执行中一起更新 |
| P7 | 渐进约束 | ERROR 必须修复，WARN 建议关注 |
| P8 | 运行态验证 | 编译通过只是起点，必须有环境 + 日志证据 |
| P9 | 对抗覆盖 | 独立 Agent 生成边界数据做对抗测试 |
| P10 | 出题解题分离 | 避免自我验证盲区 |

## 常见问题

**Q：项目很小（只有几个文件），也需要 Harness 吗？**

A：Part 1 的 AGENTS.md + 约束文档 + Linter 是最低配置，几乎零成本但收益明显。执行计划和自反馈工作流可以等项目长到需要部署的阶段再加。

**Q：没有 CI/CD 或测试环境怎么办？**

A：自反馈工作流的"部署"环节可以替换为本地启动服务 + 本地发请求。关键不是一定要有线上环境，而是"必须走到运行态验证，不能只看代码"。

**Q：对抗测试 Agent 和需求 Agent 必须是不同的 AI 会话吗？**

A：是的。核心诉求是"出题者不知道实现细节"，如果在同一个会话里，Agent 会基于自己写的代码出"能通过"的测试，失去对抗意义。

**Q：Linter 检查项怎么确定？**

A：原则是"你的约束文档里每一条'禁止'都应该有对应的自动化检查"。如果约束文档写了但 Linter 没覆盖，那条约束就只能靠 Agent 的"自觉"，早晚会失效。

**Q：适用于哪些 Agent 平台？**

A：Harness 是方法论，不绑定平台。只要你的 Agent 能读 Markdown 文件、能执行 shell 命令、能接受规则注入，就可以用。

---

## 一句话总结

> 给 Agent 一套可机器解析的约束文档让它不跑偏，一条可自动执行的验证链路让它不盲写，一个独立的对抗角色让它不自欺。这就是 Harness。

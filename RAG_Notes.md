# RAG（Retrieval-Augmented Generation）完整知识笔记


---

## 一、RAG 概述

### 1.1 什么是 RAG

RAG（检索增强生成）是一种将**外部知识检索**与**大语言模型生成**相结合的技术范式。核心思路是：用户提问后，先从知识库中检索相关文档片段，再将这些片段作为上下文注入 Prompt，让 LLM 基于真实数据生成回答。

RAG 解决的核心问题：LLM 的知识截止、幻觉（Hallucination）、以及无法访问私有数据。

### 1.2 RAG 的演进路线

**Naive RAG → Advanced RAG → Modular RAG**

- **Naive RAG**：最基础的"检索-拼接-生成"流程。文档切块 → 向量化存储 → 用户查询向量化 → 相似度检索 Top-K → 拼入 Prompt → LLM 生成。问题在于检索质量不稳定，对复杂查询处理能力弱。

- **Advanced RAG**：在 Naive RAG 基础上引入多项优化技术，包括 Query 改写、混合检索、Rerank 重排序、上下文压缩等，显著提升检索精度和生成质量。

- **Modular RAG**：将 RAG 拆解为可插拔的功能模块（检索器、生成器、路由器、评估器等），根据场景灵活组合。支持多轮迭代检索、自适应检索策略、Agent 驱动的动态决策等高级模式。

### 1.3 RAG 全链路 Pipeline

```
文档 → 切分(Chunking) → 向量化(Embedding) → 存入向量数据库
                                                    ↓
用户查询 → Query改写 → 向量化 → 检索(Retrieval) → 重排序(Rerank) → 拼接Prompt → LLM生成 → 评估
```

## 二、文档切分（Chunking）

### 2.1 为什么需要切分

LLM 的上下文窗口有限，且检索需要细粒度匹配。将长文档切成合适大小的块（chunk），既能提高检索精度，又能控制 Token 消耗。切分质量直接影响下游检索和生成效果，是 RAG 中最容易被低估但影响最大的环节之一。

### 2.2 切分策略详解

#### 策略一：固定大小 + Overlap

按固定字符数（如 512 tokens）切分，相邻块之间保留一定重叠（如 50-100 tokens）。Overlap 的作用是防止语义在切分边界处被截断。

- **优点**：实现简单，处理速度快
- **缺点**：完全不考虑语义边界，可能把一句话切成两半
- **适用场景**：快速原型验证、对质量要求不高的场景

#### 策略二：按标点/段落递归切分（Recursive Splitting）

这是 LangChain 等框架的默认策略。核心思想是**按优先级递归尝试不同分隔符**：先尝试按 `\n\n`（段落）切分，如果某块仍然超长，再按 `\n`（换行）切分，还超长就按 `.`（句号）切分，最后才按字符数硬切。

```
分隔符优先级："\n\n" → "\n" → "。" → "，" → 按字符数
```

- **优点**：尽量保持段落和句子完整性，实现成本低
- **缺点**：依赖文档格式规范，对格式混乱的文档效果一般
- **适用场景**：大多数普通文档的默认选择，性价比最高

#### 策略三：Markdown / 结构化切分

利用文档自身的结构标记（Markdown 标题 `#`、HTML 标签、代码块等）进行切分。每个标题下的内容作为一个独立块，天然保持了主题完整性。

- **优点**：切分结果语义完整，主题明确
- **缺点**：要求文档本身有良好的结构标记
- **适用场景**：技术文档、Wiki、API 文档等结构化内容

#### 策略四：语义切分（Semantic Chunking）

对文档中相邻句子计算 Embedding 相似度，当相似度出现显著下降（即"语义断裂点"）时进行切分。

```
句子1 ←→ 句子2: 相似度 0.92
句子2 ←→ 句子3: 相似度 0.88
句子3 ←→ 句子4: 相似度 0.45  ← 语义断裂，在此切分
句子4 ←→ 句子5: 相似度 0.91
```

- **优点**：切分结果语义最连贯
- **缺点**：计算成本高（每对相邻句子都要算 Embedding），速度慢
- **适用场景**：**不会全量使用**，仅针对召回质量差的问题文档做定向优化。实际工程中，大部分文档用递归切分即可，只有发现某些文档检索效果不好时，才对这些文档单独启用语义切分

### 2.3 实际工程建议

实际项目中的切分策略选择：

- **默认方案**：递归切分（Recursive Splitting），覆盖 80% 以上的普通文档
- **结构化文档**（Markdown、HTML）：用结构化切分，按标题层级拆分
- **问题文档**：对召回质量不好的文档，针对性地使用语义切分
- **Chunk 大小**：通常 256-1024 tokens，需要根据具体场景实验调优
- **Overlap**：通常设为 chunk 大小的 10%-20%

### 2.4 表格处理

表格是 RAG 中的难点，因为表格的语义依赖行列结构，普通切分会破坏这种结构。四种主要处理策略：

| 策略 | 做法 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 整表入库 | 将整个表格（含表头）作为一个 chunk | 保留完整结构 | 大表可能超长 | 小表格 |
| 逐行转自然语言 | 每行转成一句话，如"产品A的价格是99元，库存500" | 向量检索友好 | 丢失全局视角 | 需要查具体行的场景 |
| 摘要+原表 | LLM 生成表格摘要用于检索，命中后返回原始表格 | 兼顾检索和完整性 | 需要额外 LLM 调用 | 中等大小表格 |
| Text-to-SQL | 将表格存入数据库，用 LLM 生成 SQL 查询 | 支持复杂聚合查询 | 实现复杂 | 大型结构化数据 |

## 三、元信息（Metadata）

### 3.1 为什么需要元信息

每个 chunk 除了文本内容本身，还应该携带元信息（metadata）。元信息的作用是：在向量相似度之外提供额外的过滤和路由维度，大幅提升检索精度。

### 3.2 常见元信息字段

元信息需要定义统一的 Schema，团队内所有文档按同一标准写入：

```json
{
  "source": "技术方案文档",
  "doc_id": "doc_20240301_001",
  "title": "用户认证模块设计方案",
  "author": "张三",
  "created_at": "2024-03-01",
  "updated_at": "2024-06-15",
  "department": "后端架构组",
  "doc_type": "技术方案",
  "tags": ["认证", "OAuth", "SSO"],
  "chunk_index": 3,
  "total_chunks": 12,
  "heading_path": "系统设计 > 认证流程 > OAuth2.0",
  "language": "zh-CN",
  "version": "v2.1",
  "access_level": "internal"
}

```

关键字段说明：

- **source / doc_id**：溯源用，知道 chunk 来自哪个文档
- **doc_type / tags**：分类和路由，支持按类型过滤
- **heading_path**：记录 chunk 在文档中的层级位置，帮助理解上下文
- **created_at / updated_at**：时间过滤，优先返回最新内容
- **department / access_level**：权限控制和命名空间隔离

### 3.3 元信息过滤方式

元信息在向量数据库中以结构化字段存储，过滤方式取决于数据库实现：

**Pre-filtering（检索前过滤）**：先用元信息条件缩小候选集，再在子集内做向量搜索。优点是精确，缺点是如果过滤条件太严可能导致候选集太小。大多数向量数据库（Milvus、Qdrant、Weaviate）默认采用这种方式。

**Post-filtering（检索后过滤）**：先做全量向量搜索得到 Top-K，再用元信息条件过滤。优点是不会遗漏相似结果，缺点是过滤后数量可能不足 K 条。

**实际使用**：Pre-filtering 更常用，因为它能显著减少搜索空间，提升性能。查询时通过类似 SQL WHERE 的语法指定过滤条件：

```python
# Milvus 示例
collection.search(
    data=[query_vector],
    filter='doc_type == "技术方案" and department == "后端架构组"',
    limit=10
)
```

### 3.4 命名空间隔离与路由

在多租户或多业务场景下，用元信息实现命名空间隔离：不同业务线的文档存在同一个向量数据库中，但通过 `namespace` 或 `tenant_id` 字段隔离，查询时只搜索对应命名空间。

还可以基于元信息做**查询路由**：根据用户问题的意图，自动选择搜索哪个文档类型或哪个知识库。

## 四、文本向量化（Embedding）

### 4.1 向量化的完整流程

将一段文本转换为固定维度的稠密向量，完整流程分三步：

```
原始文本 → Tokenization（分词） → Transformer 编码 → Pooling（池化） → 固定维度向量
```

#### 第一步：Tokenization

将文本拆分为 Token 序列。现代模型普遍使用子词分词（BPE、WordPiece），一个中文字通常对应 1-2 个 token。分词后每个 token 映射为一个初始向量（Token Embedding），加上位置编码（Positional Encoding）后送入 Transformer。

#### 第二步：Transformer 编码

Token 序列经过多层 Transformer（Self-Attention + FFN），每个 token 的向量在注意力机制下融合了上下文信息。输出是一个**token 级别的向量序列**，形状为 `[seq_len, hidden_dim]`，比如 `[128, 768]`。

#### 第三步：Pooling（池化）

Transformer 输出的是每个 token 的向量，但我们需要**整段文本的一个向量**。Pooling 就是将 token 级向量聚合为句子级向量的操作：

- **Mean Pooling（均值池化）**：对所有 token 向量取平均。**这是当前最主流的方式**，效果稳定，被 Sentence-BERT、BGE 等主流模型采用
- **CLS Pooling**：取 `[CLS]` token 的向量作为整句表示。BERT 原始设计用的方式，但在句子相似度任务上效果不如 Mean Pooling
- **Last Token Pooling**：取最后一个 token 的向量。**不常用**，仅在 Decoder-only 架构（如 GPT）中使用，因为 Decoder-only 模型的注意力是单向的，只有最后一个 token 能"看到"前面所有内容

### 4.2 Embedding 模型选型

对于中文 RAG 场景，推荐 **BGE-M3**（BAAI 智源出品）：

- 支持三种向量表示：Dense（稠密向量）、Sparse（稀疏向量，类似 BM25 的学习版）、ColBERT（多向量交互）
- 原生支持中文，多语言能力强
- 支持最长 8192 tokens 的输入
- 一个模型同时产出稠密和稀疏向量，天然支持混合检索

其他选择：OpenAI text-embedding-3-large（英文强，中文一般）、Cohere embed-v3（多语言）、Jina Embeddings v2。

### 4.3 向量维度与存储

常见维度：768（BERT-base）、1024（BERT-large、BGE-M3）、1536（OpenAI ada-002）、3072（OpenAI text-embedding-3-large）。维度越高表达能力越强，但存储和计算成本也越高。

## 五、Query 改写（Query Rewriting）

### 5.1 为什么需要 Query 改写

用户的原始查询往往存在问题：表述模糊、过于口语化、包含歧义、或者与文档中的专业术语不匹配。Query 改写的目标是将用户查询转换为更适合检索的形式，缩小查询与文档之间的语义鸿沟。

### 5.2 主要改写方法

#### Multi-Query（多查询扩展）

用 LLM 将用户的一个问题改写为多个不同角度的查询，分别检索后合并去重。

```
用户原始问题："RAG 怎么提升效果？"

LLM 改写为：
1. "RAG 系统有哪些优化策略？"
2. "如何提高检索增强生成的准确率？"
3. "RAG pipeline 的性能调优方法有哪些？"
```

每个改写查询分别检索 Top-K，最后合并所有结果并去重。**注意：Multi-Query 需要额外的 LLM 调用**（用 LLM 生成多个改写版本），会增加延迟和成本。

- **优点**：从多个角度召回，覆盖面广，减少单一查询的偏差
- **缺点**：需要多次 LLM 调用 + 多次检索，延迟较高

#### HyDE（Hypothetical Document Embeddings）

让 LLM 先根据用户问题**生成一个假设性的回答文档**，然后用这个假设文档的向量去检索，而不是用原始问题的向量。

```
用户问题："HNSW 算法的时间复杂度是多少？"

LLM 生成假设回答：
"HNSW（Hierarchical Navigable Small World）算法的查询时间复杂度为 O(log N)，
其中 N 是数据集大小。构建索引的时间复杂度为 O(N log N)..."

用这段假设回答的向量去检索 → 与真实文档的向量更接近
```

核心洞察：**问题和答案在向量空间中的分布是不同的**。用户问题的向量可能离真实文档较远，但一个"像答案的文本"的向量会离真实文档更近。

- **优点**：有效弥合问题与文档之间的语义鸿沟
- **缺点**：同样需要额外的 LLM 调用，且如果 LLM 生成的假设回答方向错误，会导致检索偏差

#### 两种方法的共同点

Multi-Query 和 HyDE **都需要额外的 LLM 调用**，都会增加延迟。区别在于：Multi-Query 是"多角度提问"，HyDE 是"先猜答案再找证据"。实际中可以根据场景选择，也可以组合使用。

## 六、向量检索（Vector Search）

### 6.1 相似度度量

向量检索的核心是计算查询向量与数据库中所有文档向量的相似度。最常用的度量方式：

- **余弦相似度（Cosine Similarity）**：计算两个向量夹角的余弦值，范围 [-1, 1]，值越大越相似。**最常用**，因为它只关注方向不关注长度，对归一化后的向量效果好
- **内积（Inner Product / Dot Product）**：如果向量已经 L2 归一化，内积等价于余弦相似度
- **欧氏距离（L2 Distance）**：值越小越相似，适合某些特定场景

### 6.2 ANN 近似最近邻算法

精确计算查询向量与所有文档向量的相似度（暴力搜索）在大规模数据下不可行。ANN（Approximate Nearest Neighbor）算法通过牺牲少量精度换取数量级的速度提升。

#### HNSW（Hierarchical Navigable Small World）

基于图的索引算法，构建多层跳表式的图结构：

```
第3层（最稀疏）：  A -------- D
                   |          |
第2层：            A --- B -- D --- E
                   |    |    |    |
第1层：            A-B-C-D-E-F-G-H
                   （最稠密，包含所有节点）
```

查询时从最顶层开始，在稀疏图中快速定位大致区域，然后逐层下降到更稠密的层进行精细搜索。类似于"先看地图找到城市，再找街道，最后找门牌号"。

- **查询复杂度**：O(log N)
- **优点**：查询速度快，召回率高
- **缺点**：内存占用大（需要存储图结构），构建索引慢
- **适用场景**：对查询速度和精度要求高、内存充足的场景。**最主流的 ANN 算法**

#### IVF（Inverted File Index）

基于聚类的索引算法。先用 K-Means 将所有向量分成若干个聚簇（cluster），每个聚簇有一个中心点（centroid）。查询时先找到最近的几个聚簇中心，只在这些聚簇内搜索。

```
聚簇1: [v1, v5, v8, v12, ...]   中心点 c1
聚簇2: [v2, v3, v9, v15, ...]   中心点 c2
聚簇3: [v4, v6, v7, v11, ...]   中心点 c3
...

查询：先找最近的 nprobe 个聚簇中心 → 只在这些聚簇内暴力搜索
```

- **优点**：内存占用相对小，构建速度快
- **缺点**：聚簇边界处的向量可能被遗漏
- **参数**：`nlist`（聚簇数量）、`nprobe`（查询时搜索的聚簇数量，越大越精确但越慢）

#### IVF-PQ（IVF + Product Quantization）

在 IVF 基础上加入乘积量化（PQ）压缩。将高维向量拆分为多个子空间，每个子空间独立量化为一个码本 ID，大幅减少存储空间。

```
原始向量 (768维) → 拆分为 8 个子空间 (每个96维) → 每个子空间量化为 1 字节
存储：768 × 4 bytes = 3072 bytes → 压缩为 8 bytes
```

- **优点**：存储压缩比极高（可达 100x），适合超大规模数据
- **缺点**：量化带来精度损失
- **适用场景**：十亿级别数据，内存有限的场景

#### LSH（Locality-Sensitive Hashing）

通过哈希函数将相似的向量映射到同一个桶中。查询时只在同一个桶内搜索。

- **优点**：实现简单，理论优雅
- **缺点**：实际效果不如 HNSW 和 IVF，工程中用得较少
- **适用场景**：学术研究、特定低维场景

### 6.3 ANN 算法选型建议

| 数据规模 | 推荐算法 | 理由 |
|----------|----------|------|
| < 100万 | HNSW | 速度快、精度高、内存可接受 |
| 100万-1亿 | IVF + HNSW 混合 | 平衡速度和内存 |
| > 1亿 | IVF-PQ | 必须压缩，否则内存放不下 |

## 七、向量数据库选型

### 7.1 三大阵营

向量数据库的选型可以分为三个阵营：

#### 插件派：在现有数据库上加向量能力

- **pgvector（PostgreSQL 插件）**：如果团队已经在用 PostgreSQL，加个插件就能支持向量搜索。优点是运维成本低、事务支持好、SQL 生态完整。缺点是大规模向量搜索性能不如专用数据库。适合数据量 < 500 万且已有 PG 基础设施的团队。

#### 混合搜索派：传统搜索引擎扩展向量能力

- **Elasticsearch（8.x+ 版本）**：原生支持 dense_vector 字段和 ANN 搜索（基于 HNSW）。最大优势是**天然支持混合检索**——BM25 关键词搜索和向量搜索在同一个引擎内完成，不需要额外的融合层。适合已有 ES 基础设施、需要混合检索的团队。

#### 专用派：专门为向量搜索设计的数据库

- **Milvus**：开源，分布式架构，支持十亿级向量。支持多种索引类型（HNSW、IVF、IVF-PQ 等），支持标量过滤、分区、多租户。社区活跃，中文文档完善。适合大规模生产环境。

- **Qdrant**：Rust 编写，性能优秀。支持丰富的过滤条件（payload filtering），API 设计优雅。支持内存映射（mmap），可以处理超出内存的数据。适合对性能和 API 体验有要求的团队。

- **Weaviate**：Go 编写，内置模块化的向量化能力（可以直接在数据库端调用 Embedding 模型）。支持 GraphQL 查询。适合希望简化架构的团队。

- **Pinecone**：全托管云服务，零运维。按用量付费，开箱即用。适合不想自己运维、快速验证的场景。缺点是数据在第三方、成本不可控。

- **Chroma**：轻量级，Python 原生，适合本地开发和原型验证。不适合生产环境大规模使用。

### 7.2 选型决策路径

```
数据量 < 100万 且已有 PostgreSQL？ → pgvector
需要混合检索 且已有 ES？ → Elasticsearch
数据量 > 1000万 且需要分布式？ → Milvus
追求性能和 API 体验？ → Qdrant
不想运维？ → Pinecone
本地开发/原型验证？ → Chroma
```

核心原则：**能用现有基础设施解决就不引入新组件**。很多团队一上来就选 Milvus，但如果数据量只有几十万，pgvector 完全够用，还省了运维成本。

## 八、混合检索（Hybrid Retrieval）

### 8.1 为什么需要混合检索

向量检索擅长语义匹配（"汽车"能匹配到"轿车"），但对精确关键词匹配较弱（搜索特定错误码、产品型号时可能漏掉）。关键词检索（BM25）擅长精确匹配，但不理解语义（"汽车"匹配不到"轿车"）。混合检索将两者结合，取长补短。

### 8.2 两路召回架构

实际工程中，混合检索基本都是**两路召回**的架构：

```
用户查询
  ├── 路线1：向量检索（语义匹配）→ 召回集 A（如 Top-50）
  └── 路线2：BM25 关键词检索（精确匹配）→ 召回集 B（如 Top-50）
                    ↓
              RRF 融合排序
                    ↓
              最终 Top-K 结果
```

两路独立召回，各自返回自己的 Top-N 候选集，然后通过融合算法合并排序。

### 8.3 BM25 关键词检索

BM25（Best Matching 25）是经典的关键词检索算法，基于词频（TF）和逆文档频率（IDF）：

- 一个词在当前文档中出现越多（TF 高），相关性越高
- 一个词在所有文档中出现越少（IDF 高），区分度越高
- 还考虑文档长度归一化，避免长文档天然占优

BM25 的优势在于对精确关键词匹配非常可靠，且不需要 GPU、不需要 Embedding 模型，计算成本低。

### 8.4 RRF 融合排序（Reciprocal Rank Fusion）

RRF 是最常用的多路召回融合算法。核心思想：**不看分数，只看排名**。

公式：

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

其中 `k` 是常数（通常取 60），`rank_i(d)` 是文档 d 在第 i 路召回中的排名。

示例：

```
文档 X：向量检索排名第 3，BM25 排名第 7
RRF_score(X) = 1/(60+3) + 1/(60+7) = 0.0159 + 0.0149 = 0.0308

文档 Y：向量检索排名第 1，BM25 未召回
RRF_score(Y) = 1/(60+1) + 0 = 0.0164

文档 X 的 RRF 分数更高，排在 Y 前面（因为两路都召回了）
```

RRF 的关键特性：

- **基于排名而非分数**：不同检索方式的分数量纲不同（余弦相似度 0-1，BM25 可能 0-30），直接比较没有意义。RRF 只看排名，天然解决了分数归一化问题
- **候选集是并集**：两路召回的结果取并集，某个文档只在一路中出现也会参与排序，只是分数会低一些
- **支持多路扩展**：不限于两路，可以扩展到三路、四路召回，公式直接累加即可
- **k 值的作用**：k 越大，排名靠后的文档惩罚越小，结果越"平均"；k 越小，排名靠前的文档优势越大

## 九、重排序（Rerank）

### 9.1 为什么需要 Rerank

初始检索（向量检索 / BM25）追求的是**召回率**——尽量把相关文档都找出来，但排序不一定精确。Rerank 的目标是对初始召回的候选集进行**精细排序**，把最相关的文档排到最前面。

### 9.2 Bi-Encoder vs Cross-Encoder

这是理解 Rerank 的核心概念：

#### Bi-Encoder（双塔模型）—— 用于粗排/初始检索

Query 和 Document **分别独立编码**为向量，然后计算向量相似度。

```
Query  → Encoder → query_vec  ─┐
                                ├→ cosine_similarity → 分数
Doc    → Encoder → doc_vec   ─┘
```

- **优点**：Document 的向量可以预计算并缓存，查询时只需编码 Query 并做向量搜索，速度极快
- **缺点**：Query 和 Document 之间没有交互，无法捕捉细粒度的语义关系
- **角色**：用于初始检索阶段（粗排），从百万级文档中快速筛出 Top-100

#### Cross-Encoder（交叉编码器）—— 用于精排/Rerank

将 Query 和 Document **拼接在一起**送入 Transformer，让模型在内部通过 Attention 机制充分交互。

```
[CLS] Query [SEP] Document [SEP] → Transformer → 相关性分数
```

- **优点**：Query 和 Document 的每个 token 都能互相 Attend，语义理解更深入，排序更精确
- **缺点**：每个 (Query, Document) 对都要过一遍完整的 Transformer，无法预计算，速度慢
- **角色**：用于 Rerank 阶段（精排），对 Top-100 候选重新排序，选出 Top-10

### 9.3 多级漏斗架构

实际 RAG 系统中，检索和排序是一个**多级漏斗**：

```
全量文档（百万级）
    ↓ Bi-Encoder 向量检索（粗排）
Top-100 候选
    ↓ Cross-Encoder Rerank（精排）
Top-10 最终结果
    ↓ 送入 LLM 生成回答
```

这个架构的设计逻辑是：每一级用更精确但更慢的模型，处理更小的候选集。粗排追求速度和召回率，精排追求精度。

### 9.4 常用 Rerank 模型

- **Cohere Rerank**：商用 API，效果好，开箱即用
- **BGE-Reranker**（BAAI 智源）：开源，支持中文，有 v2 版本
- **Cross-Encoder from sentence-transformers**：基于 Hugging Face 的开源实现
- **Jina Reranker**：支持多语言

## 十、RAG 评估体系

### 10.1 评估的核心挑战

RAG 系统的评估比单纯评估 LLM 更复杂，因为需要同时评估**检索质量**和**生成质量**，以及两者的协同效果。

### 10.2 RAGAS 评估框架

RAGAS（Retrieval Augmented Generation Assessment）是目前最主流的 RAG 评估框架，定义了四个核心指标：

#### 检索质量指标

**Context Recall（上下文召回率）**：检索到的文档是否覆盖了回答问题所需的所有信息。

```
Context Recall = 被检索到的相关信息点数 / 回答所需的总信息点数
```

高召回率意味着检索没有遗漏关键信息。这是 RAG 最核心的指标——如果相关文档都没检索到，后面的生成再好也没用。

**Context Precision（上下文精确率）**：检索到的文档中，真正相关的占比是多少。

```
Context Precision = 检索结果中相关文档数 / 检索结果总数
```

高精确率意味着没有引入太多噪声文档。噪声文档会干扰 LLM 生成，导致回答偏离。

#### 生成质量指标

**Faithfulness（忠实度）**：LLM 的回答是否忠实于检索到的上下文，有没有编造信息。

```
Faithfulness = 回答中能从上下文推导出的陈述数 / 回答中的总陈述数
```

这是衡量幻觉的关键指标。Faithfulness 低意味着 LLM 在"瞎编"。

**Answer Relevance（回答相关性）**：LLM 的回答是否切题，是否真正回答了用户的问题。

```
评估方式：用 LLM 从回答反向生成问题，计算生成问题与原始问题的相似度
```

### 10.3 Golden Dataset（黄金测试集）

评估 RAG 系统需要一个高质量的基准测试集，即 Golden Dataset：

- 由**人类专家**标注，包含问题、标准答案、以及对应的相关文档
- 覆盖各种问题类型：简单事实查询、多跳推理、对比分析、时效性问题等
- 定期更新，反映知识库的变化

构建 Golden Dataset 的成本高，但它是评估 RAG 系统的基石。没有 Golden Dataset，所有的评估都是主观的。

### 10.4 评估实践

实际评估流程：

1. **构建 Golden Dataset**：人类专家标注 100-500 个问答对
2. **自动化评估**：用 RAGAS 框架跑自动评估，得到四个维度的分数
3. **A/B 测试**：对 RAG 系统的改动（如更换切分策略、调整 Rerank 模型），用 Golden Dataset 做 A/B 对比，看指标是否提升
4. **持续监控**：上线后持续收集用户反馈，定期用新数据更新 Golden Dataset

核心指标优先级：**Context Recall > Faithfulness > Context Precision > Answer Relevance**。召回率是地基，召回不到就什么都没有。

## 十一、Rust 在 Agent 基础设施中的趋势

近年来，Agent 和 AI 基础设施领域出现了明显的 Rust 化趋势（如 Qdrant、LanceDB、Swiftide 等）。原因包括：

- **性能**：Rust 的零成本抽象和无 GC 特性，在向量计算、索引构建等 CPU 密集型任务中性能接近 C++，远超 Python/Go
- **内存安全**：所有权系统在编译期消除内存错误，对于需要长期稳定运行的基础设施至关重要
- **并发能力**：async/await + tokio 生态提供高性能异步运行时，适合高并发的检索服务
- **WASM 支持**：Rust 编译为 WebAssembly 的能力，使得同一套代码可以在服务端和浏览器端运行
- **Python 互操作**：通过 PyO3 可以轻松为 Python 提供高性能绑定，兼顾性能和易用性

这个趋势的本质是：**AI 应用层用 Python 快速迭代，AI 基础设施层用 Rust 追求极致性能**。类似于 Web 领域 JavaScript（应用层）和 C++/Rust（浏览器引擎）的分工。
